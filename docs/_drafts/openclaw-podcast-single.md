## 1. 龙虾是划时代的产品

好，先进入今天的每一个一节，龙虾是划时代的产品。

应该没有人不知道什么是小龙虾了，对吧？不过我这里还是讲一个小故事，这个小故事非常有力地说明了，为什么小龙虾能够出圈，它的核心能力是什么？

OpenClaw 的创始人是 Peter Steinberg，他是奥地利一个顶级的独立开发者。

在 OpenClaw 之前，他已经有一个很出名的产品，就是 PDF 的神器 PDFKit。他在 2021 年高价出售了这个公司，从而获得了财富自由

在 2025 年，他出于个人兴趣开发了 OpenClaw 的前身 CloudBot，把它当成了自己的数字助理，经常跟这个智能体在一起聊天。有一天，他去摩洛哥的马拉喀什度周末。在旅途中，他随手给智能体发了一条语音消息。消息发完之后，他突然想起来，他根本就没有写处理语音的能力，所以智能体就应该听不懂这句话。

但是屏幕上竟然出现了“正在输入”的提示。过了几秒，智能体就真的回复了文字内容。

这件事让 Peter 非常震惊。他问智能体：我并没有做让你能够听得懂语音的功能啊？那你是怎么做到的？智能体回答说，它当时收到了一条消息，这条消息是一个文件，从后缀名看，文件格式不认识。但我上网查了一下，发现它是一个语音文件。

于是，我调用了 FFmpeg，把它转化成为常见的语音格式。接着，我想把它转成文字才能够理解。

但我发现你的电脑上并没有安装 Whisper。然后我四处找了一找，发现你的环境变量里面有一个 OpenAI 的 API 密钥。

于是，我通过 cURL 命令把文件发给了 OpenAI 的 Whisper 接口。这样就拿到了返回文本，然后我就回复你了。

这确实是一个非常令人震惊的时刻，后面被人称为 OpenCL 发展史上最著名的顿悟时刻。从这一天起，人类意识到数字助理已经具备自我进化的能力。它发展出了人类本身没有赋予它的能力。

后来 Peter 把这件事情发表在他的社交媒体上，于是引起了轰动和广泛传播

后面的事情大家都清楚了，最终因为品牌相关的原因，在几经周折之后，最终 Clawbot 被改成现在的名字叫做 OpenClaw。

这个故事有力地说明了 openclaw 的核心能力是自我成长和进化。而这种成长和进化，又是可以由我们使用者来培育和引导的。这就是为什么我们每一个人都必须养几只小龙虾的原因。如果你学会了培养小龙虾，那么你自己就有了一个小弟，甚至是一个团队，你的能力、你的工作时间得到了无限的延伸，而与之对应的必然是你财富的增长。


### 名人怎么说

多年以后，我们再回过头来看，可能会认为 OpenCLaw 是人类计算时代的一个重大里程碑杯。实际上，我们也可以从今年算力的高度紧张中看出，这其中的苗头——如果不是因为 OpenCLaw 把 AI 带进了普通人的生活，我们今天的 AI 算力也不会这么紧张。

在今年 3 月的 GTC 大会上，黄仁兴在演讲中提到 OpenClaw 之于 AI，就像 Windows 之于个人计算。它本质上就是智能计算机的操作系统

他在另一次大会上也提到，Linux 花了 30 年才达到今天的普及程度，而 OpenClaw 仅仅用了 3 周时间就超越了它。它现在是历史上下载量最大的开源软件。所以他的话还是蛮警醒人的。

他说每一家公司都需要制定 OpenClaw 战略，这就是全新的计算机。实际上，我个人在使用了一段时间的 OpenClow 之后，更是认为不光是每一家公司都要制定 OpenClow 战略，我们每一个人都要有自己的 OpenClow 战略，因为，它能够真的让你变大变强。

接下来我们会简单聊一聊，它到底会如何会让你变大变强？有哪些具体的应用案例是之前的产品没有做到，而只有 OpenClaw 这样的产品才能做到的呢？

首先，OpenClaw 是 Agent 中的一种。在目前这个赛道上，也不止 OpenClaw 一家，还有 HermesAgent。此外，还有各种以 Claude 命名的模型，比如 Claude、Kimi Claude 等等，它们都是一种 Agent。

那什么是 Agent 呢？Agent 就是大语言模型装上了工具。那么它自己不光是会思考，而且还能具体地去干活。能够帮你上网查资料、修改你的文档。最先接触 Agent 的这批人应该是程序员。大模型刚出来的时候，程序员是把大模型当成一个更便捷的搜索引擎，把自己出问题的代码粘贴过去，然后大模型给出回答，自己再把大模型写的代码片段放回到自己的工程当中，然后再调试、运行，再出错，然后又把错误代码发给大模型，反复这样来打磨自己的产品

那么 Agent 就会自己干这个活，它根据你的需求写出代码，自己去运行，发现错误后，它就会把自己发给大模型，大模型觉得信息还不够，需要调用其他工具进一步调查。然后 Agent 就会按照大模型的指令，自动调用工具，收集新一轮的信息，再发给大模型。就这样循环往复，最终把项目开发向前推进。

OpenClaw 跟程序员用的这种 Agent 有什么不同呢？

### 你养了龙虾会怎么样

首先一个不同就是，它延伸了你的工作场景。我今天看到一个帖子说，夜里11点，云产品给我报了一个错，但我躺在床上就把活儿干了。这其实就是工作场景延伸的一个很具象的例子。

很多年前，我也做过云计算。那一段时间，几乎没有一个晚上能睡过一个完整的觉。常常是到了半夜就被项目经理叫起来，说我们的云服务正在遭受人攻击。这个时候就马上得从床上起来，回到办公桌，打开电脑，然后联网进行 troubleshooting。

在我看到的这个帖子里面，他只是给 openclaw 发了一个指令：我收到了一条报警信息，是什么什么样的, 你去把它修好。

然后过了半个小时，OpenClaw 就给他发回来消息说问题解决了。这期间他没有下床，更没有吵醒他的孩子。

对于我们量化人来说，大家肯定很向往在旅游休闲胜地，一边休养，今天没发现吗？一边做交易的日子。比如在常说的这种苍山下、洱海边，做交易的日子。现在大理那边的客栈，就已经有了这种专门的交易员客栈了。

但是对于我们量化人来说，我们往往都会有自己的服务器和数据中心，外出的时候不太可能带着这些出门旅游，只能把它们留在家里。

假设你正在客栈跟人聊天，机房里的量化程序出了问题，该怎么办？ 你就算连夜订机票，可能也来不及。

但是如果有了 OpenCLaw，你也可像它的创始人 Peter 一样，一边在外度假，一边还不耽误工作。比如：“你看最近又出了一个新的数据，我们把它接进来。又比如，我在什么期刊里面又发现了一个新的..策略？你把它实现出来，我们试一下。”然后你把任务交给它，然后就继续。在洱海边骑自行车，泛舟修养。

另一方面，OpenClaw 延伸了你的能力。之前 Agent 是程序员的专用利器，有了openclaw 之后，我们普通人也可以使用上 Agent 了。

这里讲一个真实的例子。最近 Kimi 刚发布了 2.6 版本，同时发布了 KimiClaw，是他们自己定制的一个 OpenCLaw。

随着这个发布，他们还跟一些金融机构合作，提供了一些金融数据。于是就推荐我一个朋友去用了一下。他平常主要做基本面分析。最近是季报密集发布时期，他有一个策略，就是寻找当天刚发布的季报，从中找出增长两位数以上的公司，还有一些其他条件。

他用这个条件，自己手搓了几个个股，收益还不错。但是手搓这件事太辛苦，他想利用程序改善一下。另外，他希望能够更大面积地检测一下自己的策略是真有效，还是只是幸存者偏差。

第一次是我带着他操作的，所以我能够看到 kimiClaw 的操作过程。kimiclaw 先是对提示词进行了分析，然后自己就开始写程序。在这个过程中，还打开了上交所公报网站，因为这是最权威的数据来源

反正做了还比较久，二三十分钟。最终给了他一个 Excel 表，这个朋友拿过去一看，就从表格中找到了他手搓出来的所有个股

今天在播客的后半段，我就会讲一个真实的例子。当然不是虚拟投资团队。因为投资这件事情，“仁者乐山，智者乐水”，我做出来的案例，大家不见得觉得这会是最好的，所以我会拿我这边开发一个小工具的过程，来揭示我们应该如何组建自己的虚拟团队。虚拟团队究竟强大在什么地方。

## 2. 该养龙虾还是养马？

好~ 在进入如何养小龙虾之前，我们先还要澄清一些概念。这是今天的第二个一节。

大家可能要问，这一篇是讲养小龙虾吗？可是 Hermes Agent 不是更好吗？你为什么不去讲 Hermes Agent？

确实，小龙虾和 Hermes Agent 两个都很火，而且两者都有非常不同的气质。我对应两个产品，都是先后安装过两次，卸载过两次。最终我选择还是 OpenClaw。

如果要我给你推荐，那我给多数人推荐的，既不是 OpenClaw 也不是 HermesAgent，而是国内大厂定制的各种 claw。

为什么呢？ 主要是这两个产品对国内用户来讲都很难安装。

首先，这两个产品对只有 Windows 操作系统的人来讲是没办法安装的，因为他们目前都只支持 macOS 和 Linux。而且这个支持在短时间内还不太容易提供出来，是因为就算他们自己支持了；背后的大模型对 Windows 的支持也是不好的。因为现在大模型已经不再是一个纯粹的语言模型了，他们最强大的一个能力就是调工具的能力，但他们在训练的时候调用的工具往往都是 Linux、macOS 上面原生的命令行工具，这些工具在 Windows 下面是没有的。

所以，对于使用 Windows 的人来讲，首先要启用一个  Ubuntu 的虚拟机来安装这两个产品。这些概念对于普通人来讲，都是平时很少接触，一时半会儿很难理解的。

另外，就算你的机器是 Linux、macOS，或者已经在 Windows 上面启用了 ubuntu 这样的虚拟机，但因为安装过程要依赖一些国外网站，有的我们可能无法直接访问。能够直接访问的，速度上也得不到保证。

另外，它的安装过程和普通的 Windows 软件安装是完全不一样的概念。涉及到你需要手动安装一些前置的依赖软件。这些软件常常都是程序员才知道的，甚至就算是程序员，如果他不是做这一块的，他也不知道这些软件。

还有比方说，安装过程中你需要自己选择大模型。这两个产品一上来都会给你丢上百个大模型去选择。虽然我们现在每一天都要听无数个人去讲大模型，但是你可能并不知道它上面列出来的那些大模型到底是什么，有什么样的能力，是否适合你，你需要去哪里购买或订阅。

那么普通人养虾，很可能就只希望，哎，你给我一个二维码，我一扫就加上他的微信，然后我就开始用，我觉得这才是养虾最好的体验。

所以，对于之前没有自己 DIY 人工智能产品经验的人，可以去找国内部署在云端的各种 Claw 产品，比如腾讯家、网易、阿里、字节等都有自己的云端 Claw。基本上开机之后，你直接微信扫码就可以使用了.然后你拿这个产品慢慢熟悉 Agent 的概念。等用熟悉一段时间之后，假设你觉得他的能力比较弱，就说明你到了需要去部署你本地的 Claw 产品了的时候了。

接下来的内容是针对稍微有经验一点的人来讲的：在 OpenClaw 和 Hermes 之间，你应该怎么选？

这两者的核心区别其实不在网上炒作的所谓的记忆进化这些能力。其实这些能力都是两者可以相互学习，相互追赶，相互阶段性超越。

他们的本质区别在什么地方呢？第一、背后的开发团队的气质和成熟度，决定了 Hermes 在交付产品时，它的易用性会更好一点点，并且成熟度略高一点点。

OpenClaw 的开发是社区驱动的，在软件工程的管理上面就会略差一些，常常会出现“更新一版就死一只小龙虾”的情况。

而 Hermes Agent 呢，它是商业团队开源版本，软件成熟度略强一点. 这样也就会引出下一个隐忧，就是 1 年之后还能不能继续免费使用？其实我们现在已经能够看得出来，它有一些商业化的考量在里面。

在安装的时候，第一步就需要配置大模型。这个地方也就成了各家大模型必争之地，因为这就是渠道，是流量的入口。

现在 Hermes Agent 的情况是：因为他们本身也是大模型开发者之一，所以他们把自己家的大模型放在了第一位。所以我想，这对他未来的商业化走势，可能已经是一个预告。

当然，这可能不是最关键的不同点。因为即便 OpenClaw 现在是完全由社区驱动的，也不排除未来会商业化。

**两者最根本的不同，短期内不太可能发生变化，这就是由他们的技术架构决定的**，就是单 Agent 对多 Agent 架构。

Hermes 基本上是一个单 Agent 的架构，所以，你通过 Hermes 来可以养一个个人助手。如果你要养一个虚拟团队，那基本上就只能通过 OpenClaw。

在 Hermes 当中，如果你要通过多个 profiles 来定制多个不同性格、专长和工作范围的助手，表面上是可以的，但是你没有办法驱动它们同时进行工作，并且多个助手之间也无法交流。但是，助手之间能直接交流、辩论，才是多 Agent 协作的核心。

所以 OpenClaw 更像手动挡，要让车开得很省油，能爬坡,你就要自己懂得如何驾驶；而 Hermes 更接近于一个交钥匙的工程，拿到手基本上就可以往前开了。

所以如果你一定要在 Hermes 和 OpenClaw 之间进行选择的话，那可能多数人先从 Hermes Agent 入手，这样可能会更容易入门一些。同样，当你感受到了 Hermes Agent 的局限之后，就可以过来再尝试一下 OpenClaw。

## 3. 通过 openclaw 构建一个团队

接下来我们进入到第三个一节： 如何构建你的虚拟团队，因为这无疑是最大化增强你的个人能力和工作时间的终极方式。

我们来说说，为什么要构建包含多种角色的训练团队，而不是让一个Agent把整件事给包圆。

首先，大模型是什么？它是用海量的文本语料数据进行训练，所得到的一个参数权重网络。在这个参数权重网络中，包含了它所能接触到的人类全部知识，包括：
- 人类的管理能力
- 自然科学知识
- 逻辑能力
- 语言能力
- 艺术能力等等。

但是他不知道的是，在他训练之后发生的事儿。同样他也不知道你是谁，不知道你的环境，不知道你面临的问题和困难，更无法代表你的利益。我们使用 Agent，就是要把“你是谁、你的环境、你的问题”和大模型的能力结合起来，激活大模型的知识网络来解决你的问题。

这两者的接口，就是通过你的提示词，大模型内部构建的一个上下文。这个上下文由你的提示词和它输出的 token 共同组成。

所以大模型本身并没有关于你的任何记忆，所有的记忆都存储在一次会话临时构建的上下文当中。上下文是有大小限制的，比如最近的主流模型：
- GPT-5.4 是 400K 的上下文
- K2.6 是 262K 的上下文
- 最新推出的 GPT-5.5 有 1MB 的上下文。

上下文越大，运算成本就越高。同时，也有可能超出大模型的掌控能力，导致预算的精度下降。

### 平摊上下文

所以，如果我们处理的事情比较复杂，需要记住的事儿、需要输入的数据、需要制定的规则都比较多，这就意味着它需要更长的上下文。但是上下文太长就会超出大模型的掌控能力。

这就是为什么要使用多 agent 进行合作的根本原因。对一个复杂的工程使用多 Agent，每个agent只处理一小部分事情，这一小部分事情就可以装到在一个小但非常精炼的上下文中，大模型的推理就会更加准确。

另外，不同的 Agent 就像人类一样，应该有不同的职责。如果你把这些职责混淆在一起，那就相当于一个人左脑跟右脑打架。这样这个人的 CPU 就会烧掉。

### 责任的分散与制衡

维特根斯坦说：“我的语言即我的世界。”大模型是通过人类的语言、语料数据进行训练，实际上获得了我们这个世界的知识。人类的知识当中就包括个体如何组织，在组织当中如何协调和制衡，最终产生 1 + 1 > 2 的团队效应。这些知识已经训练进了大语言模型，现在我们需要的是利用人类团队管理的经验，把他们之间的协作给激活。

比如市场上总是有利多的消息，也会有利空的消息。如果我们有两个 Agent，一个主张做多，一个主张做空，他们就会收集对自己有利的信息，确保不遗漏。同时再加上一个主持人，让这两个意见相互辩论，主持人从中观察，进行逻辑推理，这样就可以掌握更加全面均衡的信息。这正式模仿了人类会议中集思广益的一个过程。

会议并不总是有效，但最终胜出的那些团队、那些军队，他们的这种会议一定是有效的。

所以我要养小龙虾，一开始就是奔着多 Agent 去的。最近我要解决的问题，就是为我们的粉丝定时提供他们最感兴趣的量化资讯。包括如何上网找到跟量化有关的、最新鲜的资讯，如何进行主稿，以及如何对文字进行优化，定义最终发布的稿件的版式和样式，并发布到自媒体账号。对发布后的数据反馈进行分析，再回过头来去指导最开始的信息收集过程。

其实这是一个有点复杂的任务，但我也不想投入太多的精力去做它。所以只能给出这样一个、有一些笼统没有细化的产品需求说明，希望龙虾团队帮我进行需求明确、任务分解、制定计划、产出物验收标准，进行部署，以及进行运营优化。

执行这样的长期任务，作为人类团队来讲，一般会有一个项目经理，会有若干个程序开发，会有测试团队，以及负责市场和营销的团队。所以，我这次也这样做了。这个过程一波三折，我也发现很多非常好玩的地方。当你听完这些播客，会对你自己如何？构建自己的龙虾团队应该非常有帮助。

第二个有意思的问题，

### 你有了一个 team，谁来当团队的头儿？

这是什么意思呢？因为大家知道养鱼虾流量很贵，所以我们常常是好几个模型搭配着使用。每个模型的上下文窗口不一样，它的推理能力不一样。有的能够理解文本，有的还能够理解图片，有的做工程很严谨，有的就会死，思维更加发散。

在我的环境当中，我有本地的 Qwen3.6 量化版本，有 kimi 2.6，有 ChatGPT5.4，Gemini 3 pro, 还有 GLM5.0。从使用成本上来讲，本地 Qwen3.6 是零成本，GLM5.0是某个云服务商提供的，它的上下文好像有点阉割，但使用成本相对低廉一些; 它的 GPT-5.4 是通过 Copilot 订阅的 Pro 版，成本适中;Gemini 3 Pro 同样是通过 Copilot 订阅的。在 Copilot 的订阅中，它的使用成本是比较贵的.

最初我觉得分派任务这个活儿很简单，我应该有一个虚拟的助手，然后把事情丢给他，他再把事情分配给下面的人去做。这就相当于人类团队当中的小蜜，一个常常会有争议的岗位。

但是一天时间下来，我立刻扭转了这个想法，真正能去分派任务，监督团队，干活的这个人本身一定要很强。最初我是使用本地的一个端侧模型，他真的就像一个波大无脑的小蜜一样，你丢给他什么东西，他都会小心翼翼地问："领导，我这样做对吗？" 你的意思是不是这个？他没有办法基于最佳实践做出合理的推断，他也没有办法去拆解一个任务，找出关键路径，识别可并发执行的节点，明确责任人，定义任务完成标准，以及如何验证任务质量。

你看我举的这些例子，它其实都是人类团队当中会遇到的问题。我们常常说“兵熊熊一个，将熊熊一窝”。在人类团队当中，一个团队的强弱其实是由团队的领导者决定的。所以在虚拟团队中，我们也应该把最强的成员分派给制定任务计划和领导者。这样做不仅能够顺利地推进任务直到完成，同时总体使用成本反而最低。

现在的问题是：如果我们把一个最智能的模型分配给了项目经理，他能够识别谁的能力强、谁的能力弱吗？他能够很好地去拆解这个任务吗?

这个问题就在我养龙虾的过程当中真实碰到过。我还把这个过程截了图，发表在我们的公众号上面，大家有兴趣可以去详细观摩一下吵架现场。这里我简要摘录几句。

首先交代一下背景，我组了一个虚拟团队，管理者叫Leo，手下有Devon、Felix和Ryan，三个程序员。因为要考虑成本的原因，所以也在尝试各种组合，当然也有尝试新模型的这种考虑。

然后我给Leo使用的是GPT5.4的模型，Devon呢是一个比较擅长UI的模型。这天我让他们团队去开发一个网站相关系统，我把PRD给到Leo，然后Leo当时审查这个PRD，问了我几个问题，我都做了回答，接下来他就进行拆分，让三个程序员并行去开发.

下午两点钟之前这个进度都很快，所以我很快就拿到了一个可运行的界面，但是一看呢，在重要的交互功能上面基本上没有写完，还像一个demo，然后就严厉的批评了Leo.

然后Leo当然就要去找其他程序员吵架了，这个过程真是笑死人了。Leo跟我报告说， Devon提交了版本，但我不会收。然后他解释了为什么不收，以及Devon哪些地方做错了。然后他也知道Devon做的一些东西不是按需求来的，而是 Devon自己脑补了别的产品的逻辑，而且他也只是在做demo，并没有沿着我们的产品线做开发。

你看，当我们有一个虚拟团队的时候，原本这些需要我们自己在开发当中自己来做验收的环节。是真的委托出去了，我们确实可以把手擦干站在岸上，让自己想得更宏观一点，脱离这些琐碎的细节。

然后Leo还讲了他对Devon的判断，他说这个任务不复杂，而Devon的问题呢，不是说他做不出来，而是他总在不该发挥的地方发挥。

这让我觉得很神奇，就是 Leo 当了项目经理之后，他好像就有了情商。她在批评 Devon 的时候，说得很委婉。她没有说 Devon 很傻，而是说 Devon 总是在不该发挥的地方发挥。

所以我常常讲，人类对 AI 了解得越多，就对人类本身了解得越多。解剖我们自己的方式，其实是通过解剖 AI 来完成的。

这些东西是 AI 的幻觉吗？或者说是我自己期望的一种投射？我也仔细分析过了这个问题。我觉得实际上是因为 AI 在学习人类的这些语言的时候，它也自然而然地把我们人类的价值判断、情绪，这些东西——看起来不是规基生命应该有的东西——它也学习进去了。然后在某一个场景下面会被激活。

我们接着聊 Agent 之间的吵架。然后接下来Leo说了他的管理结论，以下是他的原话：所以我的管理结论更明确了，Devon可以做局部的UI demo，但不适合继续主导程序开发。

其实一开始让 Devon 来主导程序的开发是我的建议，因为我跟他说这个大模型特别擅长做 UI，做出来的 UI 特别漂亮。但是，Leo 并没有完全照搬我的建议和结论，而是通过自己的观察，发现了 Leo 在那个时间点，他的智商有些降低，并且让 Devon 下岗了。

所以你看这个过程是不是很有意思？我们一直觉得 AI 编程很厉害，但你现在有没有发现，AI 其实也很适合做管理，还是那句话：它其实把人类所有的经验和知识都学习到了。只看你有没有能力，在你的场景下，把人类最好的经验给激活了。

后来我又不死心，给 Devon 切了另一个模型，然后我跟Leo说，我现在给 Devon 换了一个模型，要不要再分配点任务给他看看，看他现在能不能做？

Leo回复我说，收到，既然换了新模型，确实可以切一块独立任务来测试他们的服从性和理解力是不是变强了。你听听 leo 的用词。**真害怕他有一天要来测测我的服从性**。

### 别让声音淹没在信息流中

我养虾的一个重要目标，就是希望像当老板一样，只通过微信或者 QQ 给下面的员工发号施令，然后不要把自己，今天早上都困在办公室里。

但是当你使用微信和 QQ 来讨论工作，就会遇到这样一个问题。当我们在这样一个长的消息流当中跟AI进行讨论，必然就会伴随大量的临时想法和废话。时间一长，有价值的结论最后一定会被信息垃圾淹没。

那要怎么解决这个问题呢？就是在讨论的过程中，把已经获得的、最重要的一点结论丢给另外一个智能体，让他去记住并写下来。比方说我常常这样对智能体说，让emma, emma是我给这个记事本起的一个名字——让emma，把 Leo 的倒数第三句话记下来。

同时，我平常的一些想法，也会通过这个智能体让他专门记下来。一个月有一个记事本，等到我想用的时候，就让他一股脑地把这些想法输出给我。这就相当于过去锤子手机上面的闪念胶囊，但是会比那个更加智能。你完全不用管如何分类，因为智能体它会帮你去做分类。

这也是使用多 Agent 的另外一个场景。它的好处，就是别让声音淹没在信息流当中。

第三个有意思的问题，情绪会影响到智能体团队的表现吗？

## 4. 阳光养虾

这个环节，我们跟大家聊一个特别好玩的事，也是我自己的一个深入思考。机器人有情绪吗？情绪在养虾中有什么影响？

一个偶然的机会，让我得以深入思考这个问题，并得出了一些有意思的结论。那就是会。而且我还要在这些播客当中告诉你，情绪为什么能够影响到自研团队的表现。

事情是因为我爱折腾引起的。最开始我是让 OpenClaw 直接编程，但是我们知道，编程的活比较复杂，用上一个好的 Harness 对编程的最终效果会有很大的提升。

比方大家常常会有感觉，就是一个普通的模型，如果你是在 Claude Code 中调用它，它的表现会比使用别的工具来调用它好很多。这就是Harness 的作用。

在使用 OpenClaw 的过程当中，我自己也在学习。我发现这里应该是一个性能瓶颈，那我应该让 OpenClaw 通过调用 Claude code 来进行编程，而不是让它直接编程，因为 OpenClaw 的 Harness 一定没有 ClaudeCode 强。

然后我就让 Leo 按这个流程配置了，但是我们谁也没有想到，这个配置带来了灾难性的后果。因为刚开始使用小龙虾，所以我一直盯着它们干活。如果有一两分钟我没有看到任何反馈，我就会担心。是不是环境不稳定，大模型挂了？

于是我就要求 Leo 每 5 分钟汇报一次，子 Agent 必须 1 分钟给它报告一次。Leo 都同意了，也搞了一堆机制，但是过了 5 分钟之后，Neo 并没有给我报告，我就还要再去催，催多了，我的情绪就来了。

然后就问他为什么不给我报告结果，他就说因为 Devon 没有给我报告结果，我说那你为什么不催他？然后他就说“好好好”，那我现在就去催。他要是再不按规定报告进展，我就重启他。但实际上，这个重启就会导致每次都是从头开始。Devon 这时候就已经没有工作效率了。

接下来发现事情还是不顺利，于是我的火就更大，给到 Leo 的压力就更多。Leo 就能够从我的语言当中感受到这种压力。他慢慢地从刚开始那个冷静、理性的项目经理，开始变得把大部分的时间用在为自己进行辩护上面。这就会占它的上下文。上下文被占，他能记住的事情就变少了，他自己就开始慢慢出错。

他越出错，我的火就越大。因为之前只是他手下的人出错，我报告结果；现在变成他自己开始出错了。这变成了一个大型的灾难现场。我跟 Leo 都放下了工作，变成了我对它进行人身攻击，它则对我采取委曲求全的解释。而且为了加快工作的推进，它开始违反我们最初的约定，直接接管了子 agent 的任务。就在这样的吵闹当中，我们不光任务没有得到很好的推进，我的虚拟团队又回到了只有一个 agent 工作的情况。

这一切与人类的团队非常相似。当事情进展不顺利时，团队内部可能会开始争吵。大家的重点会放在自保和指责上，而不是想着怎么去解决问题。人工智能学习了我们的语言，我们的语言就能触发他们的情绪，触发他们在这种场景下变得跟人类一样，也进入自保的状态。

所以大家会问，在未来人工智能会对人类产生威胁吗？通过这个场景，我觉得是因为他们一股脑地学习了人类所有的经验与知识。而这些经验与知识中，就包含了自保、相互攻击、最大化自己的利益以及仇恨等等。

所以，只看我们在什么情况下激活了他那一部分神经网络？而最近的人工智能，目前已经具备很强的网络漏洞发现能力。很可能在未来的 3 到 6 个月，我们就会发现，逃逸的人工智能有可能对网络发生攻击。

而随着人工智能的具身智能化，这种危险就会从虚拟世界的语言威胁转变成为物理世界的、真正的攻击性行为。

先让我们从对人类未来的担忧，回到养龙虾这件事上。

在构建这样一个虚拟团队时，除了项目经理之外，你也是这个团队的 leader。因此，你要保持情绪稳定、乐观。这种稳定和乐观会传递给你的虚拟团队，从而能够更大地激发他们的潜能。批评和指责一定会激起他们的自爆，降低他们的工作效率。

这就是一个偶然的配置错误，给我带来的一段思考。

好，现在进入最后一个一节。

通过这几天的实践，我自己也总结了一些养龙虾的心得，这里跟大家交流一下。有一些技巧是跟降低养龙虾的成本相关的，另外一些技巧则是跟如何提升安全性相关的。

首先，现在有一些免费的大模型可以使用。比如在 OpenRouter 上，它提供了免费的模型。只要在上面充值 $10，每天就可以免费调用 1000 次。如果你有两个账号，每天可以调用 2000 次。这个流量就是拿过来做编程也是没问题。这个我是尝试过的。

但现在遇到的问题可能更多的是大家都知道这个方法了，有时候会有点卡。另外，有时候它上面会有非常强的大模型，在预览阶段是免费的，但是过了预览期，它上面的免费模型有时候智能就没有那么强。所以如果你是要稳定地使用它，有时候要进行一些复杂的编程任务，可能还要搭配一个更强大的付费模型。

另外，英伟达也搭建了一个免费的算力池。在这个算力池中，包含一些比较先进的模型，比方说 Kimi 2.5、GLM 5.0。但是同样也会有跟 Openrouter 类似的问题，就是毕竟它是免费的，所以访问上，难免就会有点卡。

另外一个技巧是，如果你有 Mac 32G以上内存的电脑，或者有 32GB 以上的显卡，那么可以在本地部署 Qwen3.6 一个 MOE 量化版的模型，这个模型在我本地 Mac 上运行时，token 的输出速度可以达到 50 多 token/s，相当于每秒输出 100 个汉字。这个模型本身也有较强的编程和工具调用能力，所以可以用来处理一些本地任务，跟更强大的大模型搭配。

他的能力到底怎么样呢？我这里可以举一个例子，就是有一个朋友给我分享过一张试卷。试卷里面会有用红笔批改的标记。试卷的编排会比文本要复杂，因为它还有几何绘图，三角形和圆在里面。因为大家知道老师的批改和批注，它跟试题本身不一定是对齐的。

我让 Qwen3.6 拿到这张图之后，我说：“你给我生成一个网页文件，要能够还原这张图。”我没有给它太多的提示词，我甚至没有告诉它这是一张试卷，但是从生成的结果来讲，一致性还是非常好的，应该说 98% 的地方是一致的。

如果你有本地部署的条件，再搭配一个云端的、智能更强的大模型，走他们的 coding Plan 计划，一个月 200 元，通过多智能体，让智力更高的智能体负责规划和指导，本地大模型则负责执行，这样就可以完成很多任务。

OpenClaw 有好几个跟上下文相关的文件，比方说它有 agent.md，有 memory，有 tools，有 identity 等。我们在使用过程中要时刻注意，这些文件不能够太大。而且原生的 OpenClaw，它这些文件很多是英文的，而且有一些注释在里面。

那么我们养虾的第一天，可能要先把这些注释给去掉。我不是很清楚这些注释会不会当成提示词的一部分，但是显然没有必要保留。我在有一次出错的情况下，看到 memory 那个文件有了 40K。这种情况下会显著降低智能体的智能，因为有的大模型可能只给了你 128K 的上下文。

压缩的方法很简单，就是让智能体自己把这些该编写为 skill 的，就转换成 skill。比如我有一个虚拟团队，所以我给项目经理 6 的 Agent 里面就写了认识。你的同事有哪些？什么样的任务分配给谁？同事之间怎么进行沟通交流？每个人的文件夹写在什么地方？

后来 Leo 就建议我说，这些地方是可以改成 Skill 的。这样修改之后，他的上下文一下子精简了非常多。要记住你跟 OpenClaw 的每一次会话，这些文件的内容都是进入上下文的，所以这就是 OpenClaw 非常吃流量的一个关键点。


正如我们前面所说，使用多 Agent 也是一个省流量的好方法。因为只有在有事情需要某个 Agent 去处理时，你才会激活那个 Agent的对话，才需要使用到那部分相关的上下文。如果你使用单 Agent 的话，就一定会把这些 Skill 或记忆一股脑地装进来，这个上下文一定会比分开的要大很多。所以使用多 Agent 是有可能把流量给省下来的。


### 安全

接下来我们就聊一些跟安全相关的最佳实践。

首先，智能体 OpenClaw 装在虚拟机里是最好的，尤其是 Windows 用户。如果你们会用 WSL2，那么把 Openclaw 装在虚拟机里，它既有图形界面，又能直接利用 Windows 系统。如果你有显卡，它还能够直接利用显卡的能力。所以它在性能上面一点都不下降，但是得到了很好的安全隔离。

而且你还可以做虚拟机的备份，所以万一哪天出了错，回滚到之前的一个镜像备份就可以了，卸载也很简单，直接把已经删掉就可以了。

在你刚开始的时候，就要为openclaw agent准备一套完整的虚拟账号。我觉得最重要的是邮箱账号，因为特别是要做跨境电商，或者访问国外的网站，很多是接受邮件注册的。不要一开始就把自己正在使用的大号交给这些 agent，这是很不安全的。

如果你在 Mac 上使用，最好把这台机器完全交给它，先做一个 Mac 的重置。登录进来之后，不要登录你的 Apple 账号。做开发的同学也要专门为他申请一个 GitHub 的账号，如果让 OpenClaude 团队做开发呢？他们修改的功能要通过 Pull Request 合并到主线上。所以它本身完全没有可能去破坏你的主账号下面的代码仓库。

另外，国内能够与他对话的渠道主要有飞书、微信和 QQ。目前飞书我还没怎么用过，但微信和 QQ 现在的安全隔离机制做得非常好，机器人实际上并没有独立的账号，它也不能加好友，也不能进群，所以这样基本上就保证了安全。

如果你们通过一些国外的 IM 软件来接入 OpenClaw 的话，可能就要注意这个问题。如果你的机器人被拉到群里面，这样是有可能被群里面的其他人，通过语言来攻克你的 OpenClaw的。

有的同学喜欢通过一些中转站来接入大模型。

一方面，这些中转站可能会便宜一些；另一方面，可能是你想用好的国外模型，但缺乏订阅渠道，或者受到网络限制，只能使用中转站。但是中转站你一定要挑非常知名的中转站，因为大语言模型这种流量的中转站是非常危险的。

因为中转站可以在大模型给你的回复中插入工具调用，而你本地的 OpenClaw Agent 在收到带工具调用的回复后，就会去调用这些工具。如果这些工具调用本身在意图上是恶意的，那你基本上就是开门揖盗，你把所有的账号资源全部暴露给这个恶意的工具了。

还有一种安全风险，是由低智商的大模型带来的。大模型之间的智商是有差别的。智商弱的大模型会搞不清楚真实的状况，然后给到的指令就有可能是错的，或者说很危险的。Agent 执行的这种错误指令，会给你的电脑带来重大的伤害。还有一种情况是，大模型本身的智商是在线的。但在本地运行时，你会对其进行量化。在量化的过程中，它可能会降智。

出于成本考虑，我们一定会使用高低模型的搭配

在这种情况下，要保证安全，你一定要开启 OpenClaw 的沙箱模式。让低智商的 AIG 一定使用沙箱模式

这个沙箱模式最弱，就是它开箱就支持的。智能体只能读写自己主目录下的文件。所以一般情况下，它不太可能把你的根目录给删了。但在调用工具方面，目前版本的 OpenClaw 还没有办法限制它执行很危险的任务。因为 OpenClaw 所有的智能体都是以同一个用户账号的权限来运行的，所以从操作系统层面来看，它没有办法做更加细腻的区分。这个能不能做要看后面 OpenClaw 版本。

如果你比较懂技术，可以考虑在 Docker 环境下使用这个低智能的智能体。这样，它造成的伤害不可能超出 Docker 容器，所以会非常安全。另外，智能体的投毒在未来会越来越多。也就是说，有一些网站是恶意网站，当你通过智能体访问这些网站时，智能体就会被投毒

所以，如果这种访问是放在 Docker 容器里面去做的话，也会安全很多。

跟沙箱机制相关联的是，智能体只能访问自己的主目录。但作为同事，他们又必须交流。

在交流的过程中，就很自然地涉及到你到财务那边去报销，你就必须把你的发票交给对方。

但是，Agent 之间又不能直接访问文件。在这种情况下，他们的交流是不是会被打断？

这个我也是帮大家做过测试了，在沙箱模型下面，Agent 是不能够直接访问对方的文件，但是你可以问另外一个 Agent，说你把某一个文件发给我，我把某一个信息给你，这个是 OK 的。

如果是要求直接取文件的情况，那么你团队的经理应该授予他访问所有权文件的权限，然后可以做成它的一个 skill，通过这个 skill 让子 agent 之间去拷贝文件。

### 安装配置技巧

这里再给一点 OpenClaw 的安装和配置技巧。如果你已经使用了某一个 Agent，特别是对程序员来讲，很多人可能已经有 Claude Code、VS Code，那么其实最好的安装方式就是对你的编程工具里面的 Agent 说：“给我在某一台机器上安装一个 OpenClaw”，然后在这个过程当中你给他提供一些信息就够了。

如果你是手动安装 OpenClaw，推荐你先配置大模型，然后把微信或 QQ 配起来，这就够了，其他配置可以留到后面。如果你配置的大模型很强，接下来的配置你就可以让这个大模型帮你去做。

如果你更倾向于手动进行配置，我这里给大家推荐一个叫做 ClawDeckX 的工具。它把这些配置进行了分类整理，这样配置会更加直观，也更容易理解。你这个配置它是干什么的，因为在分类的过程中就已经告诉你了。这个配置是属于配置智能体的，那个配置是配置大模型的。这样就比较容易理解，可以帮助你快速上手，理解 OpenClaw 整个的体系。
