但是今天的话题不局限于量化人，它应该是说，对于所有的人来说，都必须掌握的一个技能。在未来，它可能成为你最重要的一个资产，就是个人的专属龙虾，他可以给你一个贴心的数字助理，也可以给你一个虚拟团队。

在去年这个时候，我试用过一家 AI 公司的编程助手，叫做 Augment。当时觉得这个名字有点奇怪，像是“变大”、“变粗”之类的。后来我想通了，AI 不是替代人类，但它确实是你的机械装甲，确实能够让你变得更加强大。而养小龙虾，或者说塑造一个专属于你自己的 Agent，就是让你变大变强的方式。

## 龙虾是划时代的产品

应该没有人不知道什么是小龙虾了，对吧？那我这里还是简单介绍一下它的来由。OpenClaw 的创始人是 Peter Steinberg，他是奥地利一个顶级的独立开发者

在 OpenClaw 之前，他已经有一个很出名的产品，就是 PDF 的神器 PDFKit。他在 2021 年高价出售了这个公司，从而获得了财富自由

在 2025 年，他出于个人兴趣开发了 OpenClaw 的前身 CloudBot，把它当成了自己的数字助理，经常跟这个智能体在一起聊天。有一天，他去摩洛哥的马拉喀什度周末。在旅途中，他随手给智能体发了一条语音消息。消息发完之后，他突然想起来，他根本就没有写处理语音的能力，所以智能体就应该听不懂这句话。

但是屏幕上竟然出现了“正在输入”的提示。过了几秒，智能体就真的回复了文字内容。

这件事让 Peter 非常震惊。他问智能体：我并没有做让你能够听得懂语音的功能啊？那你是怎么做到的？智能体回答说，它当时收到了一条消息，这条消息是一个文件，从后缀名看，文件格式不认识。但我上网查了一下，发现它是一个语音文件。

于是，我调用了 FFmpeg，把它转化成为常见的语音格式。接着，我想把它转成文字才能够理解。

但我发现你的电脑上并没有安装 Whisper。然后我四处找了一找，发现你的环境变量里面有一个 OpenAI 的 API 密钥。

于是，我通过 cURL 命令把文件发给了 OpenAI 的 Whisper 接口。这样就拿到了返回文本，然后我就回复你了。

这确实是一个非常令人震惊的时刻，后面被人称为 OpenCL 发展史上最著名的顿悟时刻。从这一天起，人类意识到数字助理已经具备自我进化的能力。它发展出了人类本身没有赋予它的能力。

后来 Peter 把这件事情发表在他的社交媒体上，于是引起了轰动和广泛传播

后面的事情大家都清楚了，最终因为品牌相关的原因，在几经周折之后，最终 Clawbot 被改成现在的名字叫做 OpenClaw。


### 名人怎么说

多年以后，我们再回过头来看，可能会认为 OpenCLaw 是人类计算时代的一个重大里程碑杯。实际上，我们也可以从今年算力的高度紧张中看出，这其中的苗头——如果不是因为 OpenCLaw 把 AI 带进了普通人的生活，我们今天的 AI 算力也不会这么紧张。

在今年 3 月的 GTC 大会上，黄仁兴在演讲中提到 OpenClaw 之于 AI，就像 Windows 之于个人计算。它本质上就是智能计算机的操作系统

他在另一次大会上也提到，Linux 花了 30 年才达到今天的普及程度，而 OpenClaw 仅仅用了 3 周时间就超越了它。它现在是历史上下载量最大的开源软件。所以他的话还是蛮警醒人的。

他说每一家公司都需要制定 OpenClaw 战略，这就是全新的计算机。实际上，我个人在使用了一段时间的 OpenClow 之后，更是认为不光是每一家公司都要制定 OpenClow 战略，我们每一个人都要有自己的 OpenClow 战略，因为，它能够真的让你变大变强。

接下来我们会简单聊一聊，它到底会如何会让你变大变强？有哪些具体的应用案例是之前的产品没有做到，而只有 OpenClaw 这样的产品才能做到的呢？

首先，OpenClaw 是 Agent 中的一种。在目前这个赛道上，也不止 OpenClaw 一家，还有 HermesAgent。此外，还有各种以 Claude 命名的模型，比如 Claude、Kimi Claude 等等，它们都是一种 Agent。

那什么是 Agent 呢？Agent 就是大语言模型装上了工具。那么它自己不光是会思考，而且还能具体地去干活。能够帮你上网查资料、修改你的文档。最先接触 Agent 的这批人应该是程序员。大模型刚出来的时候，程序员是把大模型当成一个更便捷的搜索引擎，把自己出问题的代码粘贴过去，然后大模型给出回答，自己再把大模型写的代码片段放回到自己的工程当中，然后再调试、运行，再出错，然后又把错误代码发给大模型，反复这样来打磨自己的产品

那么 Agent 就会自己干这个活，它根据你的需求写出代码，自己去运行，发现错误后，它就会把自己发给大模型，大模型觉得信息还不够，需要调用其他工具进一步调查。然后 Agent 就会按照大模型的指令，自动调用工具，收集新一轮的信息，再发给大模型。就这样循环往复，最终把项目开发向前推进。

OpenClaw 跟程序员用的这种 Agent 有什么不同呢？

### 你养了龙虾会怎么样

首先一个不同就是，它延伸了你的工作场景。我今天看到一个帖子说，我的云产品报错了，我躺在床上就把它修好了。这其实就是工作场景延伸的一个很具象的例子。

很多年前，我也做过云计算。那时候有一段时间，几乎没有一个晚上能睡过一个完整的觉。常常是到了半夜就被项目经理叫起来，说我们的云服务正在遭受人攻击。这个时候就马上得从床上起来，回到办公桌，打开电脑，然后联网 training。

在我看到的这个帖子里面，他只是给 openclaw 发了一个指令：我收到了一条报警信息，是什么什么样的你去把它修好。

然后过了半个小时，OpenClaw 就给他发回来消息说问题解决了。这期间他没有下床，也没有吵醒他的孩子。

对于量化人来说，大家肯定很向往这种苍山下、洱海边，一边休闲一边做交易的日子。而且在大理那边的客栈，还有专门的交易员客栈。但是如果你缺了大米，你运行在自己机房里面的量化程序出了问题怎么办？所以如果有了 OpenCLaw，你也可像它的创始人 Peter 一样，也可以在外度假，每天给它发指示。比如：“你看最近又出了一个新的数据，我们把它接进来。我在什么样的期刊里面又发现了一个新的策略？你把它实现出来，我们试一下。”然后你把任务交给它，然后就继续。在洱海边骑自行车，泛舟修养。

另一方面，OpenClaw 延伸了你的能力。之前 Agent 是程序员的专用利器，而程序员也只能用它来进行开发工作。开发工作之外的数据工具，这些专用的 Agent 是没有或者不会去碰的。比方说做量化交易，你必须要有金融的数据库。那么直到 Kimi 发布 Kimi Claw 之前，Agent 是接触不到这些金融数据的，所以即使是程序员，也不太可能用编程工具去做投资。

对于普通人来讲，我们的投资往往属于主观交易投资。做主观交易的人很孤独，你没有办法跟人交流，因为做主观交易的人必须有自己的独立分析，不盲从，这往往导致你听不进去别人的意见。所以你也没有办法吸收他人的观点跟看法。但是对 Agent 来讲就不一样，因为他们可以摒弃情绪，只基于事实、逻辑、数据进行分析。

而使用 OpenClaw，我们就可以组建自己的投资团队。因为 OpenClown 支持多 Agent，所谓的多 Agent 就是可以有不同的角色，比如：
- 情报收集员
- 分析师
- 交易员
- 投资经理

他们可以在一起彼此讨论，相互辩论，最终有可能得出比你个人更贴近市场的投资结论。

今天在播客的后半段，我就会讲一个真实的例子。当然不是投资团队，因为投资这件事情，“仁者乐山，智者乐水”，我做出来的案例，大家不见得觉得这会是最好的，所以我会拿我这边开发一个小工具的过程，来揭示我们应该如何组建自己的虚拟团队。虚拟团队究竟强大在什么地方。

## 该养龙虾还是养马？

好在进入如何养小龙虾之前，我们先还要澄清一些概念。大家可能要问，这一篇是讲养小龙虾吗？可是 Hermes Agent 不是更好吗？你为什么不去讲 Hermes Agent？

确实，小龙虾和 Hermes Agent 两个都很火，而且两个有非常不同的气质。

实际上，如果你之前没有太多 Agent 的经验，特别是如果你不是程序员，我估计你基本上就没有太多 Agent 的这种经验

其实最好，找的是先去找国内那种部署在云端的 Claw 产品，他们的功能一定是受限制的，智力也不够强大。养到一年两年之后你一定会放弃它们，但是在这一刻可能是你上手的最好的一个产品，因为直接上手无论是 OpenClaw，还是 Hermes Agent，对于大家来讲可能多多少少都会有一些难度。

因为这两个产品从安装来讲，都会有很多难度。一是安装过程中依赖于国外的一些网站，速度上得不到保证
二是，安装过程与普通的 Windows 软件安装是完全不一样的概念，涉及到很多依赖相关的东西或者一些前提条件，这些前提条件理解起来，其实都非常偏程序员。一般人可能都不知道他们在讲什么，比方说 Nodejs 的版本，什么 bun，什么 Python，这些不要说非程序员，就是程序员他不是这个技术栈的，他其实也听不太懂。

还有比方说，你必须要自己去选择大模型，然后你可能也不知道这些大模型在哪里买，你还要记住这些大模型的型号，而且你还需要知道它们有什么样的能力。并不是所有的大模型都能拿来直接用的。

而且像 Hermes，目前还不能安装在 Windows 上，你必须要通过 Windows 的 WSL 虚拟机。我相信这些概念对普通人来讲，平常很少接触，而且大家是没有必要接触的。

那么普通人养虾，很可能就只希望，哎，你给我一个二维码，我一扫就加上他的微信，然后我就开始用，我觉得这才是深圳最好的体验。

那么国内的大厂出的这些各种 Claude，各种云端的 Claude，基本上就是这么用的。所以，在上手的体验上，直接使用云端的 claw 会是最好。

接下来的内容是针对稍微有经验一点的人来讲的：在 OpenClaw 和 Hermes 之间，你应该怎么选？

那么这两者之间有区别吗？实际上它们还是有一些区别的。很难说谁比谁更强，但他们俩确实有着非常不同的气质。OpenClaw 更像是一辆手动挡的车，你要用好它，必须懂得很多细节。而 Hermes 更接近于一个交钥匙的工程，拿到手基本上就可以往前开了。Hermes 背后的团队工程化的能力会更突出，OpenClaw 用起来，有时候确实会不太省心。

从网上的宣传来看，Hermes 被描述为具有更强的自我进化能力。但其实这一点，两者并没有本质上的差异。或者说彼此之间存在不可跨越的鸿沟，这个是不存在的。到播客的后面，我们会进入到具体的案例分析当中。你就会看到我使用 OpenClaw 的时候，OpenClaw 它自己是怎么进化的。本质上，Hermes 也没有本地的大模型，也就是说它没有本地的智能。在没有本地智能的情况下，它的自我进化也无非是靠一些 skill 定时任务。所以，它们之间很难拉开比分。

另一方面，很多人会说 OpenClaw Hermes 的网关功能更强。我觉得这其实跟大家没有太大关系。对我们来讲，只要接了微信、接了 QQ，或者接了飞书，再或者其他网关，其实你可能也用不上。

我是两个产品都比较深度的用过了。在我放弃 Hermes 的那一刻，Hermes 确实存在这个缺陷：当你通过微信使用多 Agent 时，它本质上只能有一个 Agent 活在前台

所以，当你正在使用的这个 Agent 陷入了长时间的思考和工作时，它就无法回应你的消息。而你又没办法切换到别的 Agent。那有时候，其实是因为这个 Agent 出错了、卡死了，你基本上就废掉了。当然，这是我当时发现这个问题时，大模型给我的一个回答。具体是不是这么个情况，或者说现在 Hermes 是不是还是这个情况？也就不太清楚

另外一个，就算是你使用 Hermes 遇到这个情况，其实我现在想起来还是有办法解决的。那就是你在绑定一个 QQ 时，通过双通道的方式，你至少有两个活跃的镜像。当一个死掉时，你通过另外一个命令让网关重启，这样也就救回来了

所以，即使是使用 Hermes，也是能够做到机器人永远在线的。

所以在现阶段，我觉得两个产品还没有真正拉开的那种断层领先。更多是看你有没有时间、有没有耐心去调教它。正所谓你有多强，你的虾就有多强。

所以担心自己养 OpenClaude，如果 OpenClaude 的进化能力没有 Hermes 强，那么今天可以从我的播客里面学到不少技巧，能够让你的 OpenClaude 跟 Hermes 一样强，因为本质上它们就是一些定时任务和 skills。因为两者都没有自带智能。

## 你需要一个团队

你为什么需要多 Agent? 仅仅是为了时髦？

### 平摊上下文

上下文比较贵，因为上下文涉及到服务器那边内存分配。大模型一般给的上下文窗口都比较小，大一点的也就是接不来，现在是一兆。那么如果使用多agent的话，每个agent只处理一小部分事情，这一小部分事情就可以装到它的上下文里面去，这样模型就会更智能一点。

### 责任的分散与制衡

使用多agent模型会更智能一点，然后我们还可以做到什么呢？就是可以仿照人类的团队，有的专门负责做规划，有的负责做执行，有的负责做检验。那么这，之前由于上下文和注意力机制的问题，如果你把所有的这些责任都放到一个agent里面，它的上下文都不住。从而也不专业。

现在分散成为多个agent，他们都会为自己的目标而坚持。这样编排做好了，你的事情在执行过程中方向就不会跑偏，最终的结果就会出现一个可交付的结果。

### 你有了一个 team，谁来当团队的头儿？

最初我觉得分派任务这个活儿很简单，我应该有一个虚拟的助手，然后把事情丢给他，他再把事情分配给下面的人去做。这就相当于人类团队当中的小蜜，一个常常会有争议的岗位。

但是一天时间下来，我立刻扭转了这个想法，真正能去分派任务，监督团队，干活的这个人本身一定要很强。最初我是使用本地的一个端侧模型，他真的就像一个博大无脑的小蜜一样，你丢给他什么东西，他都会小心翼翼地问："领导，我这样做对吗？" 你的意思是不是这个？他没有办法基于最佳实践做出合理的推断，他也没有办法去拆解一个任务，找出关键路径，识别可并发执行的节点，明确责任人，定义任务完成标准，以及如何验证任务质量。

你看我举的这些例子，它其实都是人类团队当中会遇到的问题。那你可能会想，你养一个虚拟的项目经理团队能做好这些活吗？他能知道谁的能力强，谁的能力弱，他能拆除任务的关键路径吗？

对，其实这个问题就在我养龙虾的过程当中真实碰到过。我把这个过程截了图，发表在我们的公众号上面，大家有兴趣可以去详细观摩一下吵架现场。这里我简要摘录几句。

首先交代一下背景，我组了一个虚拟团队，管理者叫Leo，手下有Devon、Felix和Ryan，三个程序员。因为要考虑成本的原因，所以也在尝试各种组合，当然也有尝试新模型的这种考虑。

然后我给Leo使用的是GPT5.4的模型，Devon呢是一个比较擅长UI的模型。这天我让他们团队去开发一个网站相关系统，我把PRD给到Leo，然后Leo当时审查这个PRD，问了我几个问题，我都做了回答，接下来他就进行拆分，让三个程序员并行去开发.

下午两点钟之前这个进度都很快，所以我很快就拿到了一个可运行的界面，但是一看呢，在重要的交互功能上面基本上没有写完，还像一个demo，然后就严厉的批评了Leo.

然后六当然就要去找其他程序员吵架了. Devon提交了版，但我不会收。然后他解释了为什么不做，以及哪些地方做错了。然后他也知道Devon做的一些东西，他不是按需求来的，他是自己脑补了别的产品的逻辑，而且他也只是在做demo，并没有沿着我们的产品线做开发。

然后他还讲了他对Devon的判断，他说这个任务不复杂，而他的问题呢不是说他做不出来，而是他总在不该发挥的地方发挥，我觉得这个很神，而且就是他讲的还很委婉。

然后接下来他说了他的管理结论，这四个字是他的原话，所以我的管理结论更明确了，Devon可以做局部的UI demo，但不适合继续主导spec对其实现。

那其实让Devon来主导是我的建议，因为我跟他说这个大模型特别擅长做UI，做出来UI特别漂亮。

然后Leo就说那我的建议是现在把主导权切给Ryan.

所以你看这个过程是不是很有意思？后来我又不死心，给他们切了另一个模型，然后我跟他说，我现在给 Devon 换了一个模型，要不要再分配点任务给他看看，看他现在能不能做？

然后Leo回复我说收到，既然换了新模型，确实可以切一块独立任务来测试他们的服从性和理解力是不是变强了。你听听 leo 的用词。真害怕他有一天要来测测我的服从性.



### 别让声音淹没在信息流中

我养虾，我就只想通过微信或者QQ来跟他交流。但无论使用哪一个，当我们在这样一个长的消息流当中跟AI进行讨论，必然就会伴随大量的试错、临时想法和废话。时间一长，有价值的结论最后一定会被信息垃圾淹没。

微信消息就像声音，想要回过头去查找什么内容，只能像倒带一样重听语音。这跟我们传统的视觉面板不一样，传统软件的UI是视觉化的，可以一目十行扫视跳查。

有一个文化差异，做跨境电商的人常常会有体会。在工作中，老外使用电子邮件会比即时通讯软件多一些。但是我们在国内常常用微信来交代事情。这两者有什么不同呢，就是同样是信息流，但是邮件是多线程的，你会有多个会话主题，而微信是单线程的。

### 同事间交流

Openclaw 的agent有沙箱机制，这个沙箱机制默认情况下就只限制agent只能在自己的主目录下读写。然后各个agent的能力是做高低搭配的，有的agent是程序员，会编程，有的agent是记事本，有的会利用程序员写的工具去运行，然后他们彼此之间要交换文件。

那么这个场景下怎么办呢？其实它就非常自然，就是不是让agent a去主动的去读a减b的文件，而是让a减a跟。Git的必说，你把什么什么文件发给我？

比方说为了做这些播客，我有一个agent叫emma，我就是把它当记事本来用的，然后我会让他把所有的想法按月记成一个文件，然后文件的内容格式就是日期分类以及要记录的内容，然后我就跟另外一个agent叫muse，他是我们的创意主管，我就跟他说我们今天要做一些播客，你要准备大纲，艾玛那边寄了很多想法，你去找他要出来，然后他就去跟他聊天，果然就是就跟他说，那你把记得最近几天的想法全部发给我，这样艾玛就把他的文件发过来了。

这样做有什么好处？第一，文件是分类管理的，不同类别的文件有自己的工作区。第二，自己的文件自己管理。我对艾玛是没有做任何限制，所以当muse向他请求信息的时候，他就全部都给了。但是如果你需要的话，其实这一边是可以加安全阀的，就是让他拒绝某些请求，或者说拒绝从谁这边过来的请求。。
